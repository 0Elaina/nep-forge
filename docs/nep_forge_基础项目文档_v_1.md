# docs/00-project-overview.md

# NepForge 项目概述

## 1. 项目名称

**NepForge**

## 2. 项目类型

NepForge 是一个基于 **Spring Boot + Vue 3** 的前后端分离装机与数码产品社区平台，面向对电脑硬件、装机方案、数码产品参数、硬件文章和装机交流感兴趣的用户。

项目不仅关注传统台式电脑配件，也会逐步扩展到显示器、键盘、鼠标、外设、笔记本、数码产品等更广泛的消费电子产品信息。

## 3. 项目背景

在电脑硬件与数码产品领域，用户通常会面临以下问题：

1. 配件信息分散，参数对比不方便。
2. 装机配置需要手动记录，缺少统一管理工具。
3. 新手用户不了解配件之间的搭配关系。
4. 优质装机经验、硬件评测、选购建议分散在不同平台。
5. 用户希望保存自己感兴趣的文章、配件和装机单。
6. 学生开发者需要一个贴近企业真实开发流程的完整项目，用于学习、实践和求职展示。

NepForge 的目标是将“配件库、装机单、参数对比、专栏内容、社区交流、用户收藏”整合到一个完整平台中，并通过真实的工程规范模拟企业级项目开发流程。

## 4. 项目目标

### 4.1 业务目标

NepForge 计划实现以下业务能力：

1. 展示硬件与数码产品信息。
2. 支持用户浏览、筛选、搜索配件。
3. 支持配件参数详情展示。
4. 支持用户创建、编辑、保存自己的装机单。
5. 支持装机单自动统计总价和总功耗。
6. 支持文章发布、装机指南、硬件评测、社区讨论。
7. 支持文章评论和评论回复。
8. 支持点赞、收藏、收藏夹。
9. 支持用户个人主页。
10. 支持管理员维护配件、分类、文章、评论、用户等数据。
11. 为后续交互式装机教学模块预留扩展空间。

### 4.2 技术目标

项目开发不仅要实现功能，还要模拟企业级开发流程：

1. 使用 GitHub 管理代码。
2. 使用规范化分支模型进行开发。
3. 编写完整项目文档。
4. 编写数据库设计文档。
5. 编写接口文档。
6. 使用统一响应结构和全局异常处理。
7. 使用 JWT 完成登录认证。
8. 使用 RBAC 完成权限控制。
9. 使用 Redis 优化热点数据。
10. 使用 Docker Compose 模拟真实部署环境。
11. 使用 GitHub Actions 尝试自动化构建。
12. 形成可用于实习面试展示的完整项目。

## 7. 项目技术架构

## 7.1 整体架构

项目采用前后端分离架构：

```text
用户浏览器
   ↓
Vue 3 前端应用
   ↓ HTTP / HTTPS
Spring Boot 后端服务
   ↓
MySQL / Redis / 对象存储
```

## 7.2 后端架构

推荐后端技术栈：

| 类型 | 技术 |
|---|---|
| 基础框架 | Spring Boot |
| 开发语言 | Java |
| ORM | MyBatis-Plus 或 MyBatis |
| 数据库 | MySQL 8.x |
| 缓存 | Redis |
| 权限认证 | Spring Security + JWT |
| 接口文档 | Knife4j / SpringDoc OpenAPI |
| 对象存储 | MinIO / 云 OSS |
| 构建工具 | Maven |
| 部署 | Docker / Docker Compose |

## 7.3 前端架构

# docs/01-requirements-analysis.md

# NepForge 需求分析文档

## 1. 文档目的

本文档用于描述 NepForge 项目的业务需求、用户角色、功能范围、MVP 边界和后续扩展方向。

该文档将作为后续数据库设计、接口设计、前端页面设计、后端模块划分、测试计划和版本迭代的基础。

## 2. 需求背景

电脑装机和数码产品选购具有较强的信息密度。用户在选择硬件或数码产品时，需要关注品牌、型号、价格、性能参数、功耗、兼容性、使用场景和社区评价。

传统做法通常是用户在多个网站之间切换查询信息，再手动记录自己的配置方案。这种方式存在信息分散、管理困难、对比不直观、经验沉淀不足等问题。

NepForge 希望提供一个统一平台，使用户可以在一个系统中完成配件浏览、参数对比、装机单创建、文章学习、社区交流、收藏管理等行为。

## 3. 用户角色分析

## 3.1 游客

游客主要目标是浏览公开内容，了解平台价值。

典型需求：

1. 我想查看热门配件。
2. 我想查看某个配件的详细参数。
3. 我想搜索 CPU、显卡、内存等硬件。
4. 我想查看别人公开的装机方案。
5. 我想阅读装机指南和硬件文章。

## 3.2 普通用户

普通用户是平台核心用户。

典型需求：

1. 我想注册账号并登录。
2. 我想保存自己的装机方案。
3. 我想收藏感兴趣的配件。
4. 我想收藏有用的文章。
5. 我想点赞认可的内容。
6. 我想发布自己的装机经验。
7. 我想在文章下评论和回复。
8. 我想在个人主页看到自己的文章、装机单、点赞和收藏。

## 3.3 版主

版主主要负责社区内容治理。

典型需求：

1. 我想删除违规评论。
2. 我想下架违规文章。

管理员主要负责平台数据和系统管理。

典型需求：

2. 我想管理角色权限。
3. 我想维护配件分类。
4. 我想新增、编辑、删除配件。
5. 我想管理文章和评论。
6. 我想查看平台数据概览。
7. 我想查看管理员操作日志。

## 4. 功能性需求

## 4.1 用户认证需求

### 4.1.1 用户注册

描述：用户可以通过用户名、邮箱和密码注册账号。

业务规则：

1. 用户名不能为空。
2. 用户名不能重复。
3. 邮箱不能为空。
4. 邮箱不能重复。
5. 密码需要加密存储。
6. 注册成功后默认分配普通用户角色。

优先级：高。

### 4.1.2 用户登录

描述：用户可以通过账号密码登录系统。

业务规则：

1. 登录成功后返回 Token。
2. Token 用于后续接口认证。
3. 密码不能明文比对，应使用加密算法校验。
4. 被删除或被禁用用户不能登录。

优先级：高。

### 4.1.3 当前用户信息

描述：登录用户可以查看自己的基本信息。

返回内容：

1. 用户 ID。
2. 用户名。
3. 邮箱。
4. 头像。
5. 用户角色。

优先级：高。

### 4.1.4 用户资料修改

描述：用户可以修改自己的头像、用户名等资料。

## 5. 非功能性需求

## 5.1 安全性

1. 密码必须加密存储。
2. 登录接口需要防止暴力破解。
3. 权限接口必须进行身份校验。
4. 管理员接口必须进行角色校验。
5. 用户只能修改自己的资源。
6. 文件上传需要限制格式和大小。
7. Markdown 内容需要防止 XSS。

## 5.2 性能

1. 列表接口必须分页。
2. 热门文章、热门配件可以使用 Redis 缓存。
3. 点赞数、收藏数、评论数可以使用冗余字段减少实时统计压力。
4. 高频计数可以后续改为异步写回。
5. 常用查询字段需要建立索引。

## 5.3 可维护性

1. 后端按照业务模块分包。
2. 前端按照页面和业务模块拆分目录。
3. 接口返回格式统一。
4. 错误码统一管理。
5. 枚举字段统一定义。
6. 数据库表和字段需要文档说明。
7. 每个阶段保留开发日志。

## 5.4 可扩展性

1. 配件参数使用 JSON 支持不同品类扩展。
2. 用户交互表支持多种目标类型。
3. 分类表支持父子级结构。
4. 评论表支持评论回复。
5. 后续可以扩展交互式教学板块。
6. 后续可以扩展配件数据同步。
7. 后续可以扩展装机兼容性检查。

  ↓

  ↓

  ↓

用户登录

## 11. 风险分析

## 11.1 功能范围过大风险

风险：项目功能较多，初学阶段容易陷入“什么都想做，最后都没做完”。

应对策略：

1. 严格区分 MVP 和后续扩展。
2. 每个版本只完成一组核心功能。
3. 优先完成闭环，不追求一开始完美。

## 11.2 配件数据来源风险

风险：真实配件数据量大，且“最新数据”依赖外部来源。

应对策略：

1. 第一版使用管理员手动录入和种子数据。
2. 第二阶段再考虑爬虫或外部接口。
3. 数据来源设计先预留字段。

## 11.3 参数结构复杂风险

风险：不同品类参数差异很大，传统字段难以统一。

应对策略：

1. 第一版使用 `specs_json` 存储差异化参数。
2. 常用筛选字段可以后续抽取为独立字段。
3. 后续引入参数模板表。

## 11.4 权限复杂度风险

风险：RBAC、管理员、版主、普通用户权限可能导致接口逻辑混乱。

应对策略：

1. 第一版只保留三种角色。
2. 后端统一使用权限注解或安全配置。
3. 管理端接口统一放在 `/api/admin/**`。

## 11.5 前后端协作风险

风险：个人项目中容易先写前端或先写后端，导致接口不统一。

应对策略：

1. 先设计接口文档。
