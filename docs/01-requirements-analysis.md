# NepForge 需求分析文档

## 1. 文档目的

本文档用于描述 NepForge 项目的业务需求、用户角色、功能范围、MVP 边界和后续扩展方向。

该文档将作为后续数据库设计、接口设计、前端页面设计、后端模块划分、测试计划和版本迭代的基础。

## 2. 需求背景

电脑装机和数码产品选购具有较强的信息密度。用户在选择硬件或数码产品时，需要关注品牌、型号、价格、性能参数、功耗、兼容性、使用场景和社区评价。

传统做法通常是用户在多个网站之间切换查询信息，再手动记录自己的配置方案。这种方式存在信息分散、管理困难、对比不直观、经验沉淀不足等问题。

NepForge 希望提供一个统一平台，使用户可以在一个系统中完成配件浏览、参数对比、装机单创建、文章学习、社区交流、收藏管理等行为。

## 3. 用户角色分析

## 3.1 游客

游客主要目标是浏览公开内容，了解平台价值。

典型需求：

1. 我想查看热门配件。
2. 我想查看某个配件的详细参数。
3. 我想搜索 CPU、显卡、内存等硬件。
4. 我想查看别人公开的装机方案。
5. 我想阅读装机指南和硬件文章。

## 3.2 普通用户

普通用户是平台核心用户。

典型需求：

1. 我想注册账号并登录。
2. 我想保存自己的装机方案。
3. 我想收藏感兴趣的配件。
4. 我想收藏有用的文章。
5. 我想点赞认可的内容。
6. 我想发布自己的装机经验。
7. 我想在文章下评论和回复。
8. 我想在个人主页看到自己的文章、装机单、点赞和收藏。

## 3.3 版主

版主主要负责社区内容治理。

典型需求：

1. 我想删除违规评论。
2. 我想下架违规文章。
3. 我想处理用户举报。
4. 我想维护社区讨论质量。

## 3.4 管理员

管理员主要负责平台数据和系统管理。

典型需求：

1. 我想管理用户。
2. 我想管理角色权限。
3. 我想维护配件分类。
4. 我想新增、编辑、删除配件。
5. 我想管理文章和评论。
6. 我想查看平台数据概览。
7. 我想查看管理员操作日志。

## 4. 功能性需求

## 4.1 用户认证需求

### 4.1.1 用户注册

描述：用户可以通过用户名、邮箱和密码注册账号。

业务规则：

1. 用户名不能为空。
2. 用户名不能重复。
3. 邮箱不能为空。
4. 邮箱不能重复。
5. 密码需要加密存储。
6. 注册成功后默认分配普通用户角色。

优先级：高。

### 4.1.2 用户登录

描述：用户可以通过账号密码登录系统。

业务规则：

1. 登录成功后返回 Token。
2. Token 用于后续接口认证。
3. 密码不能明文比对，应使用加密算法校验。
4. 被删除或被禁用用户不能登录。

优先级：高。

### 4.1.3 当前用户信息

描述：登录用户可以查看自己的基本信息。

返回内容：

1. 用户 ID。
2. 用户名。
3. 邮箱。
4. 头像。
5. 用户角色。

优先级：高。

### 4.1.4 用户资料修改

描述：用户可以修改自己的头像、昵称、简介等资料。

可修改字段：

1. 头像（`avatar`）。
2. 昵称（`nickname`）。
3. 简介（`bio`）。

业务规则：

1. 用户名不能和其他用户重复。
2. 用户只能修改自己的资料。
3. 管理员可以管理用户状态。

优先级：中。

## 4.2 配件库需求

### 4.2.1 配件分类

描述：系统需要支持配件分类展示。

业务规则：

1. 分类支持父子级结构。
2. 顶级分类包括 CPU、主板、内存、硬盘、显卡、电源、机箱、散热器、显示器、键盘、鼠标、其他配件等。
3. 禁用分类不应在前台展示。
4. 管理员可以维护分类。

优先级：高。

### 4.2.2 配件列表

描述：用户可以查看配件列表。

筛选条件：

1. 分类。
2. 品牌。
3. 价格区间。
4. 关键词。
5. 排序方式。

排序方式：

1. 默认排序。
2. 价格升序。
3. 价格降序。
4. 最新上架。
5. 热度排序。

优先级：高。

### 4.2.3 配件详情

描述：用户可以查看单个配件的详细信息。

展示内容：

1. 配件名称。
2. 品牌。
3. 分类。
4. 价格。
5. 封面图。
6. 参数信息。
7. 点赞状态。
8. 收藏状态。

优先级：高。

### 4.2.4 配件参数对比

描述：用户可以选择多个配件进行参数横向对比。

业务规则：

1. 第一版优先支持同分类配件对比。
2. 对比字段来自 `specs_json`。
3. 不同配件缺失的字段显示为空或 `-`。
4. 用户可以从配件详情页或列表页加入对比。

优先级：中。

### 4.2.5 管理员维护配件

描述：管理员可以新增、编辑、删除配件。

业务规则：

1. 管理员才能进入配件管理页面。
2. 配件名称不能为空。
3. 分类不能为空。
4. 价格不能小于 0。
5. 参数 JSON 需要格式正确。
6. 删除优先采用逻辑删除。

优先级：高。

## 4.3 装机单需求

### 4.3.1 创建装机单

描述：登录用户可以创建自己的装机单。

业务规则：

1. 标题不能为空。
2. 每个用户可以创建多份装机单。
3. 创建时可以为空配置，也可以从配件详情页快速加入。

优先级：高。

### 4.3.2 编辑装机单

描述：用户可以编辑自己创建的装机单。

可编辑内容：

1. 标题。
2. 描述。
3. 配件列表。
4. 配件数量。
5. 是否公开。

业务规则：

1. 用户只能编辑自己的装机单。
2. 管理员可以下架违规公开装机单。
3. 已删除装机单不应出现在普通列表中。

优先级：高。

### 4.3.3 添加配件到装机单

描述：用户可以将配件加入装机单。

业务规则：

1. 同一装机单内同一配件不能重复添加。
2. 如需多件同款配件，通过数量字段处理。
3. 添加或删除配件后，需要重新计算总价和总功耗。

优先级：高。

### 4.3.4 装机单详情

描述：用户可以查看装机单详情。

展示内容：

1. 装机单标题。
2. 作者信息。
3. 配件清单。
4. 单个配件价格。
5. 配件数量。
6. 总价。
7. 总功耗。
8. 描述。
9. 点赞数。
10. 收藏数。

优先级：高。

### 4.3.5 装机单公开与私密

描述：用户可以设置装机单是否公开。

业务规则：

1. 私密装机单只有创建者本人可见。
2. 公开装机单可以被其他用户浏览、点赞和收藏。
3. 管理员可以下架违规公开装机单。

优先级：中。

## 4.4 点赞收藏需求

### 4.4.1 点赞

描述：登录用户可以对指定内容点赞。

支持对象：

1. 文章。
2. 配件。
3. 装机单。
4. 评论。

业务规则：

1. 同一用户对同一目标只能点赞一次。
2. 用户可以取消点赞。
3. 点赞后需要更新目标对象的点赞数。
4. 未登录用户不能点赞。

优先级：高。

### 4.4.2 收藏

描述：登录用户可以收藏指定内容。

支持对象：

1. 文章。
2. 配件。
3. 装机单。

业务规则：

1. 同一用户对同一目标只能收藏一次。
2. 用户可以取消收藏。
3. 收藏时可以选择收藏夹。
4. 未登录用户不能收藏。

优先级：高。

### 4.4.3 收藏夹管理

描述：用户可以创建和管理自己的收藏夹。

功能：

1. 创建收藏夹。
2. 修改收藏夹名称。
3. 修改收藏夹描述。
4. 设置公开或私密。
5. 删除收藏夹。
6. 查看收藏夹内容。

业务规则：

1. 用户只能管理自己的收藏夹。
2. 删除收藏夹时需要处理其中收藏内容。
3. 第一版可以采用“删除收藏夹时同步取消其中收藏”的策略。

优先级：中。

## 4.5 文章专栏需求

### 4.5.1 发布文章

描述：登录用户可以发布文章。

业务规则：

1. 标题不能为空。
2. 内容不能为空。
3. 内容使用 Markdown 存储。
4. 文章可以保存为草稿。
5. 文章发布后游客可浏览。
6. 管理员或版主可以下架违规文章。

优先级：高。

### 4.5.2 文章列表

描述：用户可以浏览已发布文章。

筛选条件：

1. 关键词。
2. 作者。
3. 发布时间。
4. 热度。
5. 分类。
6. 标签。

第一版可以暂时不做分类和标签，但数据库后续应补充文章分类和标签表。

优先级：高。

### 4.5.3 文章详情

描述：用户可以查看文章详情。

展示内容：

1. 标题。
2. 作者。
3. 发布时间。
4. Markdown 渲染内容。
5. 浏览量。
6. 点赞数。
7. 收藏数。
8. 评论数。
9. 评论区。

优先级：高。

## 4.6 评论需求

### 4.6.1 发布评论

描述：登录用户可以对文章发表评论。

业务规则：

1. 评论内容不能为空。
2. 未登录用户不能评论。
3. 被删除文章不能评论。
4. 评论成功后需要更新文章评论数。

优先级：高。

### 4.6.2 回复评论

描述：用户可以回复其他评论。

业务规则：

1. 第一版只支持二级评论。
2. `parent_id = 0` 表示文章下的一级评论。
3. `parent_id != 0` 表示回复某条评论。
4. 被删除评论不允许继续回复。

优先级：中。

### 4.6.3 删除评论

描述：用户、版主和管理员可以删除评论。

业务规则：

1. 用户可以删除自己的评论。
2. 版主可以删除违规评论。
3. 管理员可以删除任意评论。
4. 删除采用逻辑删除。

优先级：中。

## 4.7 用户主页需求

### 4.7.1 个人主页

描述：用户可以访问自己的主页或其他用户公开主页。

展示内容：

1. 用户头像。

4. 用户发布的文章。

优先级：中。

### 4.7.2 我的主页

描述：登录用户可以查看自己的完整数据。

展示内容：

1. 我的资料。
2. 我的文章。
3. 我的装机单。
4. 我的点赞。
5. 我的收藏。
6. 我的收藏夹。

优先级：高。

## 4.8 后台管理需求

### 4.8.1 后台 Dashboard

描述：管理员进入后台后可以看到数据概览。

展示内容：

1. 用户数。
2. 配件数。
3. 文章数。
4. 装机单数。
5. 评论数。
6. 近期新增数据。

优先级：中。

### 4.8.2 用户管理

描述：管理员可以管理用户。

功能：

1. 用户列表。
2. 用户搜索。
3. 用户详情。
4. 修改用户角色。
5. 禁用或恢复用户。

优先级：中。

### 4.8.3 内容管理

描述：管理员和版主可以维护社区内容。

功能：

1. 文章管理。
2. 评论管理。
3. 装机单管理。
4. 举报处理。

优先级：中。

## 5. 非功能性需求

## 5.1 安全性

1. 密码必须加密存储。
2. 登录接口需要防止暴力破解。
3. 权限接口必须进行身份校验。
4. 管理员接口必须进行角色校验。
5. 用户只能修改自己的资源。
6. 文件上传需要限制格式和大小。
7. Markdown 内容需要防止 XSS。

## 5.2 性能

1. 列表接口必须分页。
2. 热门文章、热门配件可以使用 Redis 缓存。
3. 点赞数、收藏数、评论数可以使用冗余字段减少实时统计压力。
4. 高频计数可以后续改为异步写回。
5. 常用查询字段需要建立索引。

## 5.3 可维护性

1. 后端按照业务模块分包。
2. 前端按照页面和业务模块拆分目录。
3. 接口返回格式统一。
4. 错误码统一管理。
5. 枚举字段统一定义。
6. 数据库表和字段需要文档说明。
7. 每个阶段保留开发日志。

## 5.4 可扩展性

1. 配件参数使用 JSON 支持不同品类扩展。
2. 用户交互表支持多种目标类型。
3. 分类表支持父子级结构。
4. 评论表支持评论回复。
5. 后续可以扩展交互式教学板块。
6. 后续可以扩展配件数据同步。
7. 后续可以扩展装机兼容性检查。

## 6. MVP 范围

第一版 MVP 必须完成：

1. 用户注册登录。
2. 用户资料查看。
3. 配件分类。
4. 配件列表。
5. 配件详情。
6. 装机单创建。
7. 装机单编辑。
8. 装机单详情。
9. 点赞。
10. 收藏。
11. 收藏夹。
12. 文章发布。
13. 文章列表。
14. 文章详情。
15. 评论。
16. 基础后台管理。

第一版 MVP 不做：

1. 交互式装机教学。
2. 自动爬取最新配件。
3. 智能推荐算法。
4. 完整兼容性检查。
5. 微服务拆分。
6. 高并发架构。

## 8. 核心业务流程

## 8.1 注册登录流程

```text
用户填写注册信息
  ↓
前端校验
  ↓
后端校验用户名和邮箱是否重复
  ↓
密码加密
  ↓
创建用户
  ↓
分配默认角色 ROLE_USER
  ↓
注册成功
```

## 8.2 创建装机单流程

```text
用户登录
  ↓
创建装机单
  ↓
选择配件
  ↓
添加到装机单详情
  ↓
后端计算总价和总功耗
  ↓
保存装机单
  ↓
前端展示装机单详情
```

## 8.3 点赞流程

```text
用户点击点赞
  ↓
检查是否登录
  ↓
检查是否已经点赞
  ↓
写入 user_interaction
  ↓
更新目标对象 like_count
  ↓
返回最新状态
```

## 8.4 收藏流程

```text
用户点击收藏
  ↓
检查是否登录
  ↓
选择收藏夹
  ↓
检查是否已经收藏
  ↓
写入 user_interaction
  ↓
更新目标对象 favorite_count
  ↓
返回最新状态
```

## 8.5 评论流程

```text
用户输入评论
  ↓
检查登录状态
  ↓
校验评论内容
  ↓
保存评论
  ↓
更新文章 comment_count
  ↓
返回评论信息
```

## 11. 风险分析

## 11.1 功能范围过大风险

风险：项目功能较多，初学阶段容易陷入“什么都想做，最后都没做完”。

应对策略：

1. 严格区分 MVP 和后续扩展。
2. 每个版本只完成一组核心功能。
3. 优先完成闭环，不追求一开始完美。

## 11.2 配件数据来源风险

风险：真实配件数据量大，且“最新数据”依赖外部来源。

应对策略：

1. 第一版使用管理员手动录入和种子数据。
2. 第二阶段再考虑爬虫或外部接口。
3. 数据来源设计先预留字段。

## 11.3 参数结构复杂风险

风险：不同品类参数差异很大，传统字段难以统一。

应对策略：

1. 第一版使用 `specs_json` 存储差异化参数。
2. 常用筛选字段可以后续抽取为独立字段。
3. 后续引入参数模板表。

## 11.4 权限复杂度风险

风险：RBAC、管理员、版主、普通用户权限可能导致接口逻辑混乱。

应对策略：

1. 第一版只保留三种角色。
2. 后端统一使用权限注解或安全配置。
3. 管理端接口统一放在 `/api/admin/**`。

## 11.5 前后端协作风险

风险：个人项目中容易先写前端或先写后端，导致接口不统一。

应对策略：

1. 先设计接口文档。
2. 前后端按照接口文档开发。
3. 使用统一响应结构。
4. 所有分页参数和返回格式统一。
